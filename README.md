# Circuit Conquest

02641166

Circuit Conquest is a local two-player connection strategy game played on a 7 × 7 grid.
Player One (▲) aims to link the top edge to the bottom edge with an unbroken orthogonal chain.
Player Two (■) aims to link the left edge to the right edge.
The first player to complete their crossing path wins; if the board fills without a winner the result is a draw.

## Game rules

- The board is 7 × 7 cells.
- **Player One (▲)** must connect the **top edge (row 1) to the bottom edge (row 7)** using a continuous chain of triangles.
- **Player Two (■)** must connect the **left edge (column 1) to the right edge (column 7)** using a continuous chain of squares.
- Players alternate turns. Player One always moves first.
- Each turn, a player places one marker on any empty cell.
- Only **horizontal and vertical** neighbours form a connection. Diagonal adjacency does not count.
- Once a cell is occupied it cannot be reused by either player.
- The first player to complete a path between their two target edges wins immediately.
- If every cell is filled and neither player has a complete path, the game ends in a draw.

## Features

- Local two-player gameplay with no server or account required.
- Immutable game state: every valid move returns a new state object; nothing is mutated.
- Win detection via breadth-first flood fill over orthogonal neighbours.
- Draw detection when the board is full with no winner.
- Responsive browser interface that scales from narrow mobile screens to wide desktops.
- Full keyboard navigation: arrow keys move between cells; Enter or Space places a marker.
- Screen-reader labels on every cell; turn changes, wins, and draws announced via a live region.
- Players are distinguished by symbol (▲ / ■) as well as colour; no information is colour-only.
- Restart control available at any point.
- 66 behavioural unit tests covering the complete public game-module API.
- API documentation generated by JSDoc with the docdash theme.

## Requirements

- [Node.js](https://nodejs.org/) (v18 or later) and npm — used to install dependencies, run tests, and generate documentation.
- A modern web browser (Firefox, Chrome, Edge, or Safari).

No packages need to be installed globally.

## Installation

```sh
npm install
```

## Running the game

The web application uses ES modules, so it must be served over HTTP rather than opened as a local file directly.

**Option A — VS Code with Firefox (uses the supplied debug configuration):**

Open the project in VS Code and start the **"Run Web App – Firefox"** configuration from the Run and Debug panel (`Ctrl+Shift+D`). This launches Firefox with the file-origin restriction relaxed, allowing ES modules to load from the filesystem.

**Option B — any browser, using a temporary HTTP server:**

```sh
npx http-server web-app
```

Then open <http://localhost:8080> in any browser. `npx` is included with npm and does not require a separate installation step.

## Running tests

```sh
npm test
```

Uses Mocha. The test files are discovered automatically via the pattern in `.mocharc.json`.

## Generating API documentation

```sh
npm run docs
```

Output is written to `docs/`. Open `docs/index.html` in a browser to browse the generated API reference.

## Controls

- **Click** (or **tap**) an empty cell to place a marker.
- **Enter** or **Space** — activate the focused cell.
- **Arrow keys** — move focus one cell up, down, left, or right within the board.
- **Tab** — move from the board to the Restart button (or back).
- **Restart Game** button — begin a new game at any time.

## Accessibility

- Semantic HTML throughout: `<header>`, `<main>`, `<section>`, and native `<button>` elements.
- Every board cell is a real `<button>`, reachable and activatable from the keyboard.
- **Roving tabindex:** only one cell is in the Tab sequence at any time; arrow keys navigate within the board without adding 49 separate tab stops.
- `aria-disabled="true"` on occupied and post-game cells keeps them keyboard-focusable for board exploration while preventing accidental moves.
- Each cell carries an `aria-label` such as `"Row 3, column 5, occupied by Player One, triangle"` (one-based coordinates, explicit symbol name).
- A `role="status"` live region with `aria-live="polite"` announces turn changes, wins, and draws to assistive technology.
- Visible focus indicator on every interactive element: a 3 px black outline with a yellow background on board cells.
- Players are identified by both symbol (▲ / ■) and by colour; no game information is communicated through colour alone.
- Responsive layout: the board sizes to `min(90vw, 30rem)` and maintains a 1:1 aspect ratio at all viewport widths.
- All CSS transitions are suppressed when `prefers-reduced-motion: reduce` is active.

## API design

`CircuitConquest.js` is the single source of all game rules. The browser UI (`main.js`) does not contain any game logic: it renders state, translates user input into `makeMove` calls, and reads results using the module's query functions.

Key API decisions:

- **Successful moves** return a new `GameState` object; every board row is independently copied.
- **Invalid moves** (occupied cell, out-of-bounds coordinates, non-integer coordinates, game already over) return the **original state reference unchanged** — no exception is thrown. The UI detects rejection via reference equality (`nextState === state`).
- **`getBoard()`** returns a defensive copy so callers cannot accidentally mutate internal state.
- Private helpers (`opponentOf`, `inBounds`) are marked `@private` and are not part of the public API.
- `PLAYER_ONE` and `PLAYER_TWO` are exported constants; the UI uses these rather than the raw string values.

## Testing

The 66 tests are organised into 11 describe blocks and cover:

- **Initial state** — board dimensions, all cells empty, starting player, move counter, status.
- **Legal-move validation** — in-range empty cells; occupied, out-of-bounds, and fractional coordinates.
- **Move enumeration** — `getLegalMoves` count and row-major order.
- **State transitions** — marker placement, turn alternation, move-counter increment, unchanged non-target cells.
- **State immutability** — original object properties, original board rows, and all row references remain unchanged after `makeMove`; new object and new row references are returned for valid moves.
- **Defensive copying** — `getBoard()` returns a copy that can be mutated without affecting the state.
- **Status queries** — `isGameOver`, `getWinner`, `isDraw`, `getGameStatus` in normal and game-over conditions.
- **Win detection** — straight vertical path (Player One), straight horizontal path (Player Two), indirect orthogonal path, broken path (not a win), diagonal-only chain (not a win), move rejection after a win.
- **Draw detection** — filled 2 × 2 board with no winner; no legal moves; `isValidMove` returns false.
- **Restart** — empty board, counters reset, Player One first, size preserved, repeated restart.

## Known limitations

- Local two-player only — there is no computer opponent.
- No persistent scores or match history.
- No undo functionality.
